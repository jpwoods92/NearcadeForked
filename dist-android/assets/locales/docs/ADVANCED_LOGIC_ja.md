### 1. インプットインジェクションアーキテクチャ
Nearsec は、オペレーティング システム レベルの入力インジェクションを処理するために、別の Python サイドカー プロセス (`input_driver.py`) に依存します。 Node.js サーバーは、バイナリのゲームパッド API 配列を含む WebSocket を受け取り、それらを標準の JSON 構造に解凍し、`stdin` 経由で Python にパイプします。

**Linux の「uinput」実装**
Linux では、「uinput」カーネル モジュールを利用します。従来のエミュレータでは、マウス、キーボード、およびゲームパッドの機能が 1 つの複合仮想デバイスに結合されていることがよくあります。これにより、最新のゲーム エンジン (Unreal Engine 5 や Unity など) で深刻な問題が発生し、デバイスを積極的にポーリングし、アナログ スティックのドリフトをマウスの動きと誤認して UI のちらつきを引き起こすことがよくあります。

これを解決するために、「linux_uinput.py」はデバイスを厳密に分離します。
* ゲームパッドは、「Xbox 360」 (VID `0x045e`、PID `0x028e`) または `DualSense` デバイスとして明示的に生成されます。
* キーボード/マウス イベントは、完全に別個の仮想 USB バス上で生成されます。
* ユーザーが入力プロファイルを変更すると (ゲームパッドからエミュレートされた KBM に切り替えるなど)、Python スクリプトは積極的に `old_gp.destroy()` を呼び出し、新しいプロファイルを初期化する前にカーネル内の仮想 USB ケーブルを物理的に切断します。これにより、16 個以上の無効なコントローラーが検出されたためにゲームがクラッシュする「コントローラー フラッディング」が防止されます。

### 2. オーディオ仮想化パイプライン
デスクトップ通知や Discord ボイスチャットをキャプチャせずに Linux でアプリケーション固有のオーディオをルーティングするには、PipeWire/PulseAudio グラフを直接操作する必要があります。

**ハイブリッド ループバック システム**
Node.js サーバーが起動すると、「initVirtualAudio()」によって一連の「pactl」コマンドが実行されます。
1. **モジュール Null シンク:** `NearsecAppAudio` を作成します。これはデジタル ブラック ホールとして機能します。これは、ゲームがターゲットにできる出力デバイスです。
2. **モジュール リマップ ソース:** `NearsecAppMic` を作成します。これにより、ブラック ホールのモニターが、ブラウザの「getUserMedia」 API が安全にキャプチャできる認識可能な入力デバイス (マイク) にマップされます。
3. **モジュール ループバック:** `NearsecAppAudio` シンクからホストの物理ヘッドフォンに直接、目に見えない永続的なワイヤーを作成します。
4. **ボリューム コントロール:** 「pactl set-sink-volume NearsecAppAudio 70%」を明示的に実行して、ローカル システム ボリュームと組み合わせたときにループバックによってクリッピングや聴覚障害が発生しないことを確認します。

**自動ルーティングとブラックリスト**
「server.js」の「routeGameAudio()」関数は、Patchbay ライブラリと連携して、システム上のすべてのアクティブなオーディオ ノードを読み取ります。ゲームをホワイトリストに登録する代わりに、スマート ブラックリスト (`AUDIO_BLACKLIST = ['discord', 'teamspeak', 'telegram']`) を使用します。 3 秒ごとに、音を発する新しいアプリケーション バイナリをスキャンします。ブラックリストに載っていない場合は、PipeWire 出力ノードを「NearsecAppAudio」シンクに物理的にリンクします。

### 3. WebRTC トランスポート層
Nearsec は従来のストリーミング サーバーではありません。これは、直接ピアツーピア (P2P) 接続を調整するシグナリング サーバーです。

**ICE 交渉とターン**
ほとんどのビューアは対称 NAT ルーターの背後に存在するため、直接 STUN 接続は頻繁に失敗します。 Nearsec は、OpenRelay TURN サーバーの資格情報を「RTCPeerConnection」構成に挿入することでこれを軽減します。直接の UDP パンチスルーが失敗した場合、トラフィックは TURN リレー経由で TCP ポート 443 にフォールバックし、厳格な企業や大学のネットワークでも 99% の接続成功率を保証します。

**双方向オーディオ (ボイスチャット)**
ホストのアップロード帯域幅を損なうことなく Voice-over-IP (VoIP) を実装するために、Nearsec は「スイッチボード」アーキテクチャを使用します。
* 視聴者はローカルマイクをキャプチャし、そのトラックをアウトバウンドの「RTCPeerConnection」に添付します。
* ホストはこれらのトラックを受信し、非表示の `<audio autoplay>` タグを生成します。
* ホストは、この音声を他の視聴者に再ブロードキャストしません。代わりに、ホストのローカル ブラウザは、受信 WebRTC オーディオ トラックをネイティブにミックスして物理スピーカーに出力し、無限フィードバック ループを防ぐために「NearsecAppAudio」シンクを完全にバイパスします。

### 4. ビデオキャプチャとウェイランド
Linux での画面キャプチャは断片化していることで知られています。 Nearsec は、Electron の「desktopCapturer」と最新の Chromium フラグを組み合わせて利用し、X11 と Wayland コンポジタの両方をスムーズにサポートします。

Wayland で実行している場合、Electron は画面キャプチャ リクエストをネイティブ XDG デスクトップ ポータル (`xdg-desktop-portal`) に委任します。これにより、ネイティブ OS ダイアログがポップアップ表示され、ユーザーに画面またはウィンドウを共有する許可を求めます。

このポータルには人間の介入が必要なため、固有の遅延が発生します。 Wayland ポータルがビデオ トラックを返す前にホストが WebRTC オファーを送信すると、ネゴシエーション ループがクラッシュします。これを修正するために、「viewer.js」は「Vanilla ICE」の収集を強制します。SDP Answer を送信する前に、「e.candidate === null」になるまで待機します。これにより、Wayland ポータルが PipeWire ビデオ ストリームを正常に割り当てて送信者に接続するのに十分な長さだけハンドシェイクが人為的に停止されます。

### 5. 接続状態とクリーンアップ
マルチクライアント P2P 環境の安定性には、積極的なガベージ コレクションが必要です。

**ゴーストポートの軽減**
Electron アプリが強制的に閉じられると、Node.js は孤立した「cloudflared」トンネルまたはスタックした TCP ポートを残す可能性があります。起動時に、Nearsec は「kill-port」を使用してポート 3000 をスクラブし、Express サーバーが正常にバインドできるようにします。

**孤立した仮想デバイス**
同様に、Python サイドカーが突然強制終了された場合、`uinput` デバイスは `/dev/input/` ディレクトリ内で永久にアクティブなままになります。 Node.js の `cleanup()` フックは、`SIGINT`、`SIGTERM`、および Electron の `window-close` イベントをトラップします。終了する前に、最後の `{ type: 'destroy_all' }` JSON ペイロードを Python の `stdin` に送信し、Python にすべてのコントローラーの登録を解除させます。同時に、特に起動時に保存された `loopbackModuleId` 整数を対象とした `pactl unload-module` コマンドを発行し、仮想オーディオ ケーブルを完全に破壊し、Linux オーディオ グラフをデフォルトの状態に復元します。

このプロジェクトでは、コード生成と構造計画に人工知能の大規模言語モデルを使用します。
